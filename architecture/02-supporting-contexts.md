# 02 · Supporting & Generic контексты цепи

> Модули `inference-chain/x/`: bls, collateral, streamvesting, restrictions, genesistransfer, bookkeeper.
> Назад к [индексу](../ARCHITECTURE.md) · к [ядру](01-core-proof-of-compute.md).

Все они — **исполнители**, которых дирижирует `x/inference` на границах эпох. Каждый имеет узкую ответственность и не владеет политикой «когда/сколько».

## `x/bls` — пороговая криптография для моста

**Контекст:** проводит per-epoch **Distributed Key Generation (DKG)** среди валидаторов, создавая общий BLS12-381 групповой ключ, *секрет которого не держит никто*, и предоставляет сервис **t-of-n порогового подписания** для авторизации операций cross-chain моста с доказательством согласия >50% веса валидаторов.

**Для чего (не beacon случайности):**
- (A) **Межэпоховая цепочка доверия** — валидаторы *прошлой* эпохи пороговой подписью заверяют групповой pubkey *новой* эпохи (→ `DKG_PHASE_SIGNED`).
- (B) **Мост** — `RequestThresholdSignature` вызывается из bridge-обработчиков `x/inference` для подписи **Ethereum-совместимого** payload (`abi.encodePacked → keccak256 → hashToG1`), который релеер отправляет в EVM-контракт.

**Слот-взвешенный Shamir/Feldman VSS:** секрет шарится на `I` слот-долей (живой genesis: `i_total_slots = 100`, `t_slots_degree_offset = 50`; комментарий в коде упоминает «прод 1000», но это нереализованная цель — фактически 100); валидатор владеет непрерывным диапазоном слотов ∝ весу. Порог `t = I − offset` (= 50 → нужно ≥51 слот-доли) = >50% веса. Так «t-of-n взвешенный» чисто сворачивается в «>50% веса» без отдельного слоя весов.

**Стейт-машина DKG** (по высоте блока, в `EndBlock`): `DEALING → VERIFYING → DISPUTING → COMPLETED → SIGNED` (или `FAILED`). Каждый гейт требует `slots > I/2`. Агрегат: **`EpochBLSData`** по эпохе.

**Умные приёмы:**
- **Анти-O(N²) газ:** dealer-части / вектора верификации / частичные подписи лежат под суб-ключами по отправителю (регидрируются при чтении) — N-й отправитель платит константный газ (чинит реальный баг, где поздние дилеры «дорожали» и валили DKG).
- **Агрегация** через Лагранжа-в-нуле по глобально уникальным слотам, один `blst` MSM.
- **Детерминированное on-chain разрешение споров:** обвинённый дилер публикует открытую долю + 32-байтный ECIES-seed; цепь *повторно шифрует* детерминированно и побайтово сравнивает с оригиналом, затем проверяет долю против коммитментов. Валидный ответ → исключают жалобщика; невалидный/отсутствует → исключают дилера.
- **Без слэшинга** — единственная санкция — *исключение* из набора валидных дилеров (и из группового ключа).
- **Обязательные доказательства жалобы** + `DealerValidityProof` гасят «бесплатные» голоса approve-all/reject-all.
- Хуки (`AfterThresholdSigningCompleted/Failed`) зовут назад в `x/inference` для очистки/авто-возврата мостовых операций.

## `x/collateral` — кастодиан залога и исполнитель слэшинга

**Контекст:** финансовый кастодиан + движок применения штрафов: эскроует депозиты, ведёт epoch-очередь анбондинга, исполняет пропорциональный слэшинг от имени `x/inference` и `x/staking`, не владея *политикой* когда/сколько слэшить.

**Связь с PoC 2.0 (ключевая идея) — гибридная модель веса:** PoC даёт **Potential Weight**; `Base Weight = Potential × BaseWeightRatio` (дефолт 0.2) даётся безусловно, остальной **Collateral-Eligible Weight** нужно *активировать* залогом:

```
activated = min(eligible, collateral / CollateralPerWeightUnit)
effective = base + activated
```

Залог *оценивает влияние GPU* (worked-пример ~100 gonka на H100/эпоху). **Grace-период** (до эпохи 180) трактует ratio как 1.0 (весь вес бесплатен).

**Механика и инварианты:**
- Состояние (`cosmossdk.io/collections`): `CollateralMap` (активный per-address), `UnbondingIM` (двойной индекс `(completionEpoch, addr)` + `ByParticipant`), `Jailed`, `SlashedInEpoch` (идемпотентность).
- **Вывод** убирает средства из активного залога сразу (вес падает следующей эпохой), но токены остаются в модуле, слэшируемы, до `completionEpoch = current + UnbondingPeriodEpochs`. Частичные выводы в одну эпоху агрегируются.
- **Слэшинг** (только keeper, нет `MsgSlash`): пропорционально к активному + анбондинг той же долей. **Кап по требуемому залогу** — когда `requiredCollateral>0`, цель слэша ограничена, чтобы пере-депонировавших не штрафовали сверх нужного весу. Идемпотентно per `(epoch, participant, reason)`. Слэш идёт в **gov** (не сжигается, несмотря на формулировки в proposals).
- Три триггера (политика — в inference): инвалидация (0.20), простой (0.10), консенсус-фолты (через staking-хук `BeforeValidatorSlashed`). Staking-хуки зеркалят jail-статус.

## `x/streamvesting` — вестинг наград, синхронный эпохам

**Контекст:** эскроу токенов + планировщик релиза, выдающий награды по одной «эпохе» за раз согласно per-account графику, синхронизированному с часами инференса.

**Механика:**
- Состояние: один `VestingSchedule` на аккаунт = `[]EpochCoins` (позиционный FIFO-массив; индекс 0 = следующая эпоха).
- **Математика депозита:** `base = A/N`, `remainder = A%N`; `base` в каждую эпоху, `remainder` в эпоху 0 — точное сохранение, пыль в начало. Несколько наград накладываются аддитивно, массив не растёт длиннее максимального N.
- **Релиз:** `AdvanceEpoch` (зовётся раз в эпоху из inference) снимает индекс 0 *каждого* графика, шлёт монеты, удаляет пустые. **Линейный стрим, без cliff.** Позиционный (не по номеру эпохи) — опирается на ровно один вызов за эпоху.
- API доступен только gov и модулю inference. Inference гонит WorkCoins через `WorkVestingPeriod`, RewardCoins через `RewardVestingPeriod` (180 в проде, 0 в dev = прямая выплата).

## `x/restrictions` — bootstrap-lockup переводов

**Контекст:** временный governance-контролируемый lockup пользовательских переводов на период запуска сети, сохраняя все протокольные операции, с авто-роспуском на заданной высоте.

**Механика:**
- Хук в `x/bank` через depinject-`SendRestrictionFn` (группа `"bank-send-restrictions"`), оценивается до каждого send. **Блок:** account→account. **Разрешено:** всё, что касается fee-collector или модуля (в любую сторону) — газ, стейкинг, gov-депозиты, эскроу инференса, награды текут. Fail-safe = активно, если параметры не загрузились.
- Активно при `blockHeight ∈ [0, RestrictionEndBlock)` (прод ~1 555 000 ≈ 90 дней; `0` = выключено).
- **Аварийные исключения:** governance-шаблоны `{from/to (вайлдкард "*"), maxAmount, usageLimit, expiryBlock}`. Двухуровневая авторизация: gov создаёт шаблон, любой подходящий юзер сам исполняет `MsgExecuteEmergencyTransfer`.
- `auto_unregistration.go`: в EndBlock, после end-блока, эмитит разовое `restriction_lifted` и ставит sentinel.

## `x/genesistransfer` — разовая миграция founder-аккаунтов

**Контекст:** безопасный атомарный одноразовый перенос *полного владения* genesis/преаллоцированным аккаунтом — все балансы **и** остаток графика вестинга — назначенному получателю, с неизменяемым аудит-следом «не более одного раза».

**Механика (`transfer.go`):** обычный `MsgSend` не умеет двигать заблокированные vesting-монеты и нести *график* вестинга, плюс bootstrap-restrictions блокируют переводы. Трюк: **понизить genesis-аккаунт до простого BaseAccount** (разблокировать vesting-монеты), прогнать все балансы через модуль (легитимно минуя restrictions), затем **пересобрать эквивалентный vesting-аккаунт** того же типа на получателе с *остаточным* графиком.

- Не переносится: pubkey/account-number/sequence; делегации/стейкинг.
- **Одноразовость:** `TransferRecord` по genesis-адресу; проверка до и атомарная запись с переносом. Опциональный whitelist (`AllowedAccounts` + `RestrictToList`, по умолч. выкл). Подписант = сам genesis-аккаунт.

## `x/bookkeeper` — аудит-декоратор над `x/bank` (Generic)

**Контекст:** тонкий учётно-аудиторский декоратор над `x/bank`: исполняет стандартные движения денег и эмитит структурные **лог-записи** (не on-chain состояние) с метками суб-аккаунтов и memo для off-chain аудита.

**Механика:**
- Интерфейс `BookkeepingBankKeeper` зеркалит шесть методов bank, каждый с доп. `memo`, плюс `LogSubAccountTransaction`. Модули зависят от этого интерфейса (через depinject), а не от сырого bank-кейпера.
- Каждая обёртка делегирует реальному bank, затем логирует per-coin (дебет/кредит парами в `DoubleEntry`; минт из / burn в псевдо-аккаунт `"supply"`). Логирование строго подчинено (только после успеха bank), не блокирует.
- `LogSubAccountTransaction` — *чисто виртуальная* (без движения денег) запись логических earmark'ов между суб-аккаунтами. В ходу: collateral (`collateral`, `collateral-unbonding`), streamvesting (`vesting`), inference (`settled`, `owed`, `balance`).
- On-chain состояние — пустые Params; лог транзакций **не запрашивается on-chain** (эфемерные логи узла). Намеренный logging-only дизайн.

## Расхождения «доки ↔ код»

- `x/bls`: proposal-доки описывают 3-фазный DKG; в коде есть 4-я фаза `DISPUTING` с on-chain адъюдикацией.
- `x/collateral`: слэш идёт в gov (не «burned»); legacy неиспользуемый `UnbondingKey` prefix 3.
- `x/restrictions`: коды ошибок в README (1109-1112) не совпадают с кодом (1102-1108); «unregistration» — логический флаг, не реальное снятие регистрации.
- `x/genesistransfer`: (1) `TransferStatus` gRPC-запрос ключует записи по bech32-строке, тогда как писатели — по `addr.Bytes()` → реальные записи не находятся (`grpc_query.go:29` vs `validation.go:295`); (2) `MsgTransferOwnership` возвращает пустой ответ при заявленном `transfer_record` в proto.
